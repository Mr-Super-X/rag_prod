# RAG 知识库平台 — 术语表

> 帮助非技术用户和产品人员理解平台中使用的 RAG 及技术术语。

---

## 核心概念

| 术语 | 英文 | 解释 |
|------|------|------|
| **RAG** | Retrieval-Augmented Generation | 检索增强生成。简单说就是：AI 回答问题前先去知识库中"查资料"，用查到的资料作为依据来回答，而不是凭空编造 |
| **知识库** | Knowledge Base (KB) | 你上传的文档集合。一个知识库对应一个项目或团队的全部文档 |
| **文档摄入** | Ingestion | 上传文档后平台帮你"消化"它的过程——读取文字→切分成段落→转换成数字向量→存入索引 |
| **向量 (Embedding)** | Vector Embedding | 把一段文字转换成一组数字（1024 个），语义相似的文字会得到相似的数字组合。搜索时用于找到"意思相近"的段落而非只能匹配关键词 |
| **语义搜索** | Semantic Search | 基于向量相似度搜索——不问"有没有这个词"，问"有没有意思相近的内容"。比如搜索"如何提升系统速度"，能匹配到包含"性能优化方法"的文档 |

---

## 搜索相关

| 术语 | 英文 | 解释 |
|------|------|------|
| **BM25** | Best Match 25 | 传统关键词搜索算法。考虑词的出现频率和文档长度，搜索包含相同关键词的段落。与语义搜索互补 |
| **混合检索** | Hybrid Retrieval | 向量语义搜索 + BM25 关键词搜索同时进行，两者结合比单独任何一种都准确 |
| **RRF 融合** | Reciprocal Rank Fusion | 把向量搜索和 BM25 搜索的结果按排名重新计算分数然后合并，取排名靠前的结果 |
| **精排 (Reranker)** | Reranker | 对混合检索出的候选结果用 AI 再次逐个评分，挑出最相关的片段。提升准确性但会变慢，默认关闭 |
| **Top-K** | — | 搜索返回的最相关结果数。比如 Top-K=10 表示取最相关的 10 个片段送入 AI 回答 |
| **检索兜底** | Fallback | 当检索无结果时的策略。平台会返回"抱歉，当前知识库中未找到与您问题相关的内容"，而非强行编造 |

## AI 与回答

| 术语 | 英文 | 解释 |
|------|------|------|
| **LLM** | Large Language Model | 大语言模型——就是回答问题的 AI 大脑。本项目使用 Qwen2.5-7B（本地）或 DeepSeek（云端） |
| **Embedding 模型** | Embedding Model | 专门负责将文字"翻译"成向量的 AI 模型。本项目使用 BGE-m3。理解文字时需要它，回答问题时用的是 LLM——是两个不同的模型 |
| **上下文改写** | Query Rewriting | 多轮对话时，把"那它怎么安装"这种依赖上文的问句改写成"XX 软件怎么安装"的独立问句，提高检索准确度 |
| **幻觉** | Hallucination | AI 编造不存在的信息。RAG 通过限定 AI 只用检索到的文档内容回答来减少幻觉，但不能完全消除 |
| **引用来源** | Source / Citation | AI 回答中标注的 `[1]` `[2]` 编号，点击可展开对应的原始文档片段。用于验证 AI 是否基于文档回答 |
| **上下文窗口** | Context Window | LLM 一次能处理的最大文字量。传入的检索片段 + 对话历史 + 问题不能超过这个限制 |

## 部署相关

| 术语 | 英文 | 解释 |
|------|------|------|
| **Ollama** | — | 本地运行 AI 模型的工具。让你在不联网、不需要 GPU 的情况下运行 LLM |
| **Docker Compose** | — | 一键启动多服务的编排工具。一条命令同时启动 PostgreSQL、Redis、Ollama 三个服务 |
| **Docker 镜像** | Docker Image | 一个打包好的软件运行环境，含代码、依赖、配置。相当于"虚拟机快照" |
| **Docker 数据卷** | Docker Volume | 容器中持久化数据的存储位置。重启或更新容器不会丢失数据 |
| **SSE** | Server-Sent Events | 服务器推送事件。让 AI 的回答逐字显示（流式输出），不用等全部生成完再一次性显示 |
| **JWT** | JSON Web Token | 用户认证用的加密令牌。登录后获得，之后每次请求带上它证明身份 |
| **Refresh Token** | — | 长效令牌。access token 过期后用它自动续签，不用反复登录 |

## 文档处理

| 术语 | 英文 | 解释 |
|------|------|------|
| **分块 (Chunk)** | Chunk | 文档被切分后的段落单元。每个 chunk 约 500 字，相邻 chunk 有 50 字重叠以避免切断关键信息 |
| **异步处理** | Async Processing | 文档上传后立即返回成功，后台默默处理。用户不需要等处理完成，可继续做其他操作 |
| **增量索引** | Incremental Indexing | 重新上传同名文件时，只处理内容变化的部分，跳过来自上次索引的片段。加速文档更新 |
| **Cross-Encoder 精排** | Cross-Encoder Reranking | V14: 对混合检索结果用 LLM listwise 方式二次排序，从 Top-25 候选池中选出 Top-10。CPU Ollama 下 similarityRerank（embed 批量调用）延迟不可控，禁用后直接走 LLM listwise |
| **向量迁移** | Vector Migration | V13: 切换 Embedding 模型后对知识库向量重建索引的过程。通过 progressStep 追踪进度，LanceDB 版本切换实现回滚 |
| **答案缓存** | Answer Cache | V14: Redis 存储问答结果，同问题 24h 内重复提问秒回，文档更新自动失效 |

## 版本标记

在功能清单和 CHANGELOG 中，版本号 V1-V14 对应功能的交付批次：
- **V1**：核心平台（知识库 + 文档 + 问答）
- **V2**：流式输出 + 安全加固
- **V3**：备份 + 日志 + 增量索引
- **V4**：安全防护 + 体验增强
- **V5**：云端 LLM + 深色模式
- **V6**：成员管理 UI + Agent
- **V7**：体验增强（格式/分组）
- **V8**：管理员控制台
- **V9**：对话导出 + 统计
- **V10**：引用高亮 + 文档预览 + 对话搜索
- **V11**：反馈闭环 + 企业就绪
- **V12**：体验闭环（趋势图 + 置信度 + 进度 + 失败汇聚）
- **V13**：AI 引擎配置 + 向量迁移
- **V14**：Cross-Encoder 精排 + Redis 答案缓存
