## 一、RAG 基础与架构（必问 ★★★★★）

### Q1：什么是 RAG？为什么要用它而不是直接让 LLM 回答？

**出现率：95%+ | 难度：入门**

RAG（Retrieval-Augmented Generation，检索增强生成）的核心思路是：**LLM 回答之前，先去知识库"查资料"，用查到的资料作为依据来回答**。

不用 RAG 时 LLM 有两个致命问题：
1. **幻觉**：LLM 会凭空编造不存在的信息，而且编得还很自信
2. **知识截断**：训练数据截止到某个日期，之后的信息它完全不知道

本项目用 RAG 解决了这两个问题——你把公司文档上传到知识库，提问时系统先检索出最相关的段落，再把段落内容和问题一起送给 LLM，LLM 被限定"只根据这些资料回答"。

**本项目实现位置**：`src/services/chat.service.ts`，整个流程是 `rewriteQuestion → retrieve → generate`。

### Q2：画一下你们 RAG 系统的整体架构，从文档上传到用户提问的完整数据流

**出现率：90%+ | 难度：中等**

```
┌─────────────────────────────────────────────────────┐
│                    文档摄入流程                        │
│                                                       │
│  PDF/Word/MD/TXT/XLSX/PPTX                            │

│    → splitText() 段落切分（500字块 + 50字重叠）         │
│    → embed() 向量化（BGE-m3, 1024维）                  │
│    → LanceDB 存储向量 + PostgreSQL 存chunk元数据        │
│    → Redis 倒排索引（BM25关键词检索）                    │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│                    问答流程                            │
│                                                       │
│  用户提问                                              │
│    → rewriteQuestion() 上下文改写（多轮对话指代消解）     │
│    → Agent拦截检测（时间/计算器等意图）                  │
│    → retrieve() 混合检索                               │
│        ├─ embedSingle() 向量检索 (LanceDB, Top-25)     │
│        ├─ bm25Search() 关键词检索 (Redis倒排, Top-25)   │

└─────────────────────────────────────────────────────┘
```

**面试话术**：先讲两个流程分开，再讲中间的桥梁（LanceDB向量库和Redis倒排索引是检索的数据基础）。重点强调：**索引和查询用的是两个不同的AI模型**（Embedding用BGE-m3，生成用Qwen2.5-7B），这是面试官喜欢的细节。

**本项目实现位置**：摄入 `src/services/doc.service.ts` + `src/pipeline/splitter.ts` + `embedder.ts`；问答 `src/services/chat.service.ts` + `retrieval.service.ts`。

### Q3：混合检索具体怎么做的？向量检索和 BM25 各自有什么优缺点？

**出现率：85%+ | 难度：中等**

**向量检索（语义搜索）**：把问题和文档都转成向量（1024维数字），算余弦相似度，找"意思最接近"的段落。
- 优点：能搜到意思相近但用词不同的内容（"性能优化" ↔ "提速方案"）
- 缺点：对专有名词不敏感（"Qwen2.5"可能搜不到）

**BM25（关键词搜索）**：基于词频统计和倒排索引，找"关键词匹配"的段落。
- 优点：专有名词、代码、缩写搜得准
- 缺点：同义词搜不到

**本项目混合方案**（`src/services/retrieval.service.ts`）：

```typescript
// 1. 向量检索 — LanceDB HNSW 索引
const queryVector = await embedSingle(question);
const vectorResults = await searchVectors(kbId, queryVector, topK);

// 2. BM25 — Redis 倒排索引 + jieba 中文分词
const bm25Results = await bm25Search(kbId, question, topK);

// 3. RRF 融合 (k=60)
function rrfFuse(vectorResults, bm25Results) {
  const scores = new Map();
  const k = 60;
  vectorResults.forEach((r, rank) => {
    scores.set(id, (scores.get(id) || 0) + 1 / (k + rank + 1));
  });
  bm25Results.forEach((r, rank) => {
    scores.set(chunkId, (scores.get(chunkId) || 0) + 1 / (k + rank + 1));
  });
  return scores; // 按融合分数降序排列
}
```

**RRF 公式**：`score = Σ(1 / (k + rank))`，其中 k=60。这个参数的含义是：排名差 60 名等价于位列第 1 名和第 2 名的分数差。值越大，不同排名之间的权重差异越小。

**关键设计决策**：使用 Redis SET 而非 SortedSet 存储倒排索引。查询时用管道（pipeline）批量读取，返回匹配的 chunkId 集合。评分简化为词频/总词数（`score / terms.length`），而非完整的 BM25 公式（因为我们只需要相对排序，RRF 融合后会重排）。

**面试追问**："为什么不用 Milvus/Weaviate/Pinecone？"

我们的考量：LanceDB 是嵌入式向量库，不需要单独部署服务，数据存在本地磁盘。适用于中小规模场景（<100万向量），降低运维复杂度。如果未来数据量增长，可以迁移到 Milvus。

## 二、检索优化（高频 ★★★★★）

### Q4：Reranker 精排是什么？你们怎么做的？

**出现率：80%+ | 难度：中等偏难**

Reranker 是对 RRF 融合后的候选结果（Top-25）做**二次精排**，本质是用更精确（但更慢）的方法重新排序后取 Top-10。

**本项目两种方案**（`src/pipeline/reranker.ts`）：

| 方案 | 方法 | 速度 | 精度 | 本项目状态 |
|------|------|------|------|-----------|
| similarityRerank | Embedding 余弦相似度逐条打分 | 快（GPU） | 中 | **已禁用** |
| llmListwiseRerank | LLM 一次性选出 Top-K | 慢（3-5s） | 高 | **当前使用** |

**Why similarityRerank 被禁用？** 在 CPU Ollama 环境下，BGE-m3 Embedding API 对多条文本的逐条请求延迟不可控（批量调用也可能超过 10s），测试发现还不如直接 LLM listwise。

**LLM listwise 实现**：
```typescript
// 把 25 个候选片段编号，让 LLM 选出最相关的 10 个
const prompt = `选出最相关的 10 个片段编号，逗号分隔：
${query}

[1] 片段1内容...
---
[2] 片段2内容...
...

编号：`;
// temperature=0, num_predict=50 保证输出稳定且只输出编号
```

**降级策略**：Reranker 异常 → 返回原始 Top-10 候选，不阻塞问答流程。默认 `RERANKER_ENABLED=false`，开启后会显著增加延迟（3-5s）。

**面试亮点**：这个设计权衡了精度和延迟，失败降级保证可用性，temperature=0 保证确定性输出。

### Q5：BM25 的中文分词怎么做的？面对专业术语怎么办？

**出现率：70%+ | 难度：中等**

**本项目方案**（`src/services/retrieval.service.ts`——`tokenize()`）：

```typescript
// 优先 jieba 分词，不可用时降级字符 bigram
async function getCutFn() {
  try {
    const jiebaModule = await import("@node-rs/jieba");
    // ... jieba 分词
  } catch {
    // 降级：字符 bigram + 词级切分
    return (text) => {
      // 中文：单字 + bigram 组合
      // "高性能" → ["高", "高性", "性", "性能", "能"]
      // 英文：按词切分，统一小写
    }
  }
}
```

- **优先方案**：`@node-rs/jieba`（Rust 版本，速度快）
- **降级方案**：字符级 bigram。对中文单字提取 + 相邻二字组合，英文单词按空格切分后统一小写
- **额外处理**：正则 `[^一-龥a-zA-Z0-9]` 过滤标点符号

**面试追问**："bigram 降级的效果如何？"

bigram 的召回率不如 jieba，但作为降级方案足够用。"人工智能"在 bigram 下会被切成 `["人", "人工", "工", "工智", "智", "智能", "能"]`，搜索"人工"或"智能"都能命中。缺点是会产生误匹配（"工人"也可能命中），但在 RRF 融合阶段向量检索会弥补这个缺陷。

### Q6：检索结果的引用追踪怎么做的？

**出现率：65%+ | 难度：中等**

**前端展示**（`MessageBubble.vue`）：
- LLM 回答中输出 `[1] [2]` 引用标注
- 前端 `parseContent()` 将文本拆分为 `[{type:"text", content:"..."}, {type:"citation", index:1}, ...]` 的 segment 数组
- 用 `v-for` 渲染（非 `v-html`，防 XSS）
- 点击引用编号展开/收起对应的 sources[N-1] 原文片段

**后端兜底**（`chat.service.ts`——`validateAndInjectCitations()`）：
- 验证 LLM 输出中的 `[N]` 是否对应有效的 source（索引不越界）
- 剥离无效引用编号
- **Bigram 兜底注入**：如果 LLM 完全没输出引用，基于回答文本和 source 内容的 bigram 重叠率自动注入最相关的引用

**面试亮点**：这是一个"信任但验证"的设计——LLM 不总是遵守指令输出引用，后处理保障了引用溯源功能的可靠性。

## 三、文档处理（高频 ★★★★）

### Q7：文档怎么切分？chunk 大小怎么定的？

**出现率：75%+ | 难度：中等**

**本项目方案**（`src/pipeline/splitter.ts`）：

```
参数：CHUNK_SIZE=500（字）, CHUNK_OVERLAP=50（字）
策略：按段落（双换行）切分 → 贪心合并到 500 字 → 保留 50 字重叠
```

**为什么是按段落而不是按字数硬切？** 按字数硬切可能在句子中间切断，破坏语义完整性。按段落切 + 贪心合并能保证每个 chunk 是完整的语义单元。

**为什么是 500 字？** 太小（<200 字）→ 上下文信息不足，LLM 难以理解。太大（>1000 字）→ 噪声增加，检索精度下降；LLM 上下文窗口有限。500 字是 BGE-m3 推荐的最佳区间（配合 50 字重叠防止边界信息丢失）。

**增量索引**（V3）：
```typescript
// 每个 chunk 计算 SHA256 hash
// 重新上传同名文件时：hash 相同 → 跳过；hash 不同 → 重新 embed
// 旧目标 chunk → 从 LanceDB 删除
```

**面试追问**："文档里有表格、代码块怎么处理？"

当前版本按段落切分时，表格和代码块会作为一个段落保留。这是简化处理，复杂的 Layout-aware 切分（如 LlamaIndex 的节点解析器）可以做但增加了复杂度，当前场景下不是瓶颈。

## 三-A、长文本处理与切分策略（新热点 ★★★★★）

### Q9：长文档（100 页+）你们怎么处理的？chunk 重叠策略怎么保证语义不丢失？

**出现率：80%+ | 难度：中等偏难**

这是一道"坑很多"的题。我分三层来讲：切分策略、重叠窗口、边界保护。

**第一层：切分策略——凭什么按段落而不是按字数硬切？**

```
本项目：按段落（双换行 \n\n）→ 贪心合并到 500 字 → 保留 50 字重叠
硬切方案：每 500 字一刀切，不管句子是否完整
```

按字数硬切的致命问题：可能在句子正中间切断。
```
原文："BGE-M3 是 BAAI 开源的多语言 Embedding 模型，支持中英日韩等 100+ 语言"
硬切(chunk=20字)："BGE-M3 是 BAAI 开源的多语言 " + "Embedding 模型，支持中英日韩等 100+ 语言"
按段落切："BGE-M3 是 BAAI 开源的多语言 Embedding 模型，支持中英日韩等 100+ 语言"
```
硬切后 LLM 读到第一个 chunk 会看到残缺句子，无法理解；向量化后"多语言 "这个残缺片段也没什么语义价值。

**本项目实现**（`src/pipeline/splitter.ts`——`splitText()`）：
```typescript
// 1. 按段落（双换行）分割
const paragraphs = text.split(/\n\s*\n/);

// 2. 贪心合并：逐段往后加，超过 500 字就切新 chunk
for (const para of paragraphs) {
  if (currentChunk.length + trimmed.length > CHUNK_SIZE && currentChunk.length > 0) {
    // 当前 chunk 满了 → 保存
    chunks.push({ content, contentHash: sha256(content), index, metadata });
    // 重叠：从当前 chunk 尾部保留 CHUNK_OVERLAP 字，拼到下一个 chunk 开头
    const overlap = currentChunk.slice(-CHUNK_OVERLAP);
    currentChunk = overlap + "\n\n" + trimmed;
  } else {
    currentChunk += (currentChunk ? "\n\n" : "") + trimmed;
  }
}
```

**第二层：重叠窗口——为什么是 50 字？少了/多了会怎样？**

```
┌─────────────────────────────────────────────┐
│ Chunk N-1: ...性能优化方法包括缓存和数据    │
│             库索引两个方面。                 │
│                          ←── 50字重叠 ──→   │
│ Chunk N:   库索引两个方面。缓存方面可以      │
│            使用 Redis 减少数据库查询...      │
└─────────────────────────────────────────────┘
```

- **太小（<20 字）**：关键信息正好落在 chunk 边界上，两个 chunk 都只有半句，LLM 无法理解上下文
- **太大（>150 字）**：冗余信息太多，浪费 LLM 上下文窗口（每个 chunk 多 150 字 × 10 条 = 1500 字浪费）
- **50 字**：中文大约 2-3 句话，足以覆盖一个完整语义单元。BGE-m3 的 500 字 chunk 推荐重叠比例约 10%

**关键面试金句**："chunk 重叠的本质是在精度和效率之间做 tradeoff——太大浪费上下文窗口，太小丢失边界语义。"

**第三层：段落边界保护——为什么重要？**

假设原文是：
```
## 第一章：性能优化
...200字...

## 第二章：安全配置  ← 这个标题刚好落在 chunk 末尾
...300字...
```

如果不保护标题边界，"安全配置"可能被切到上一个 chunk 或被孤立。本项目按段落切分天然保护了这种边界（标题是独立段落）。

**但要注意的反模式**：按段落切分对**无段落结构的文档**（如纯 JSON 导出、日志文件）效果差。这种场景需要按固定窗口切 + 滑动窗口。

**面试追问**："Markdown 文档里有标题层级（H1/H2/H3），你们怎么处理？"

实话实说：当前版本没有利用标题层级做结构化切分。更好的做法是 LlamaIndex 的 `SentenceSplitter` + 标题感知（给 chunk 的 metadata 加上所属章节标题），检索时可以把章节标题也拼到 context 里。这是未来的优化方向。

### Q9-A：语义切片（Semantic Chunking）是什么？和固定窗口切分有什么区别？你们为什么没用？

**出现率：65%+ | 难度：偏难 | 区分度：极高**

**语义切片**是按"语义相似度"动态决定切分边界，而不是按固定字数或段落。它的核心思路是：**相邻两句话如果"意思很远"，就切一刀；如果"意思很近"，就保留在同一个 chunk 里。**

```
固定窗口切分（本项目）：
  第 1-500 字 → chunk 1
  第 451-950 字 → chunk 2（50字重叠）
  ↑ 不管内容是什么，按字数一刀切

语义切片：
  句子相似度：0.95 0.92 0.88 0.35←切！→0.91 0.89 0.93
  ↑ 相似度骤降的地方就是"天然的分界点"
```

**原理**：对文档逐句 Embedding → 计算相邻句子的余弦相似度 → 找到相似度的"断崖"位置 → 在这些位置切分。

```typescript
// 伪代码
const sentences = splitBySentence(text);
const embeddings = await embed(sentences);  // [BGE-m3, 1024维]

let chunks = [];
let currentChunk = [sentences[0]];

for (let i = 1; i < sentences.length; i++) {
  const similarity = cosineSimilarity(embeddings[i-1], embeddings[i]);

  if (similarity < 0.5) {
    // 语义断开 — 新主题开始 → 切分
    chunks.push(currentChunk.join(" "));
    currentChunk = [sentences[i]];
  } else {
    // 语义连续 — 继续合并
    currentChunk.push(sentences[i]);
  }
}
```

**对比表**：

| 维度 | 固定窗口切分 | 语义切片 |
|------|------------|---------|
| 切分依据 | 字数（500字） | 语义相似度 |
| 主题完整性 | 可能在句子中间切断 | 按主题自然分界 |
| 实现复杂度 | 极简单（本项目 `splitter.ts` 不到 50 行） | 需要对每个句子 Embedding + 相似度计算 |
| 性能开销 | 0 | **高**——N 句话需要 N 次 Embedding |
| 适用场景 | 段落结构清晰的文档 | 无段落结构的文档（聊天记录、会议纪要） |

**本项目为什么没用语义切片？**

三个真实原因：

**① CPU 环境的性能约束**：语义切片需要对每句话做 Embedding。以一篇 50 页的 PDF（约 25000 字）为例——句子数约 500-800 句，需要 500+ 次 BGE-m3 Embedding 调用。在 CPU Ollama 上每批 5 条需要 100+ 批次，嵌入时间可能超过 3-5 分钟。相比之下，固定窗口切分（按段落拆分 + 贪心合并）是纯字符串操作，毫秒级完成。

**② 段落结构文档的效果差异不大**：项目处理的文档绝大部分是技术文档/操作手册/规章制度，这类文档**天然有段落结构**。按双换行切分已经能很好地区分主题边界（每个段落约等于一个语义单元）。语义切片的优势主要体现在"无段落结构"的文档——但这种文档在我们的场景中占比很低。

**③ 投入产出比**：一段 50 行的 splitter.ts 解决了 90% 的场景。为剩余 10% 的场景引入 10× 的实现复杂度和 100× 的性能开销，在当前阶段不合理。

**什么时候该上语义切片？**

| 信号 | 说明 |
|------|------|
| 用户大量上传会议纪要/聊天记录 | 这类文档无段落结构，固定窗口切分效果差 |
| 上了 GPU 服务器 | Embedding 延迟从秒级降到毫秒级，语义切片的性能瓶颈消除 |
| 用户反馈"AI 的回答经常断章取义" | 可能是切分边界不当导致关键信息被截断 |
| 需要支持多语言混合文档 | 固定字数的"字"在不同语言中意义不同（中文 500 字 ≈ 英文 300 词） |

**面试话术**："语义切片是更优的方案但不是最优的选择。工程决策要看场景——CPU 环境 + 技术文档为主 + 段落结构清晰 → 固定窗口切分是投入产出比最高的方案。知道语义切片的存在、理解它的优势和代价、能有理有据地解释'为什么我没用它'——这才是面试官想看到的。"

### Q10：Lost in the Middle 是什么？你们有遇到吗？怎么解决的？

**出现率：75%+ | 难度：偏难**

**Lost in the Middle** 是 LLM 的一个已被多篇论文证实的现象：当给 LLM 一个很长的上下文时，它对**开头和结尾**的信息关注度最高，对**中间位置**的信息关注度显著下降。

```
LLM 对上下文各位置的注意力分布：
  高 ████████░░░░░░░░░░░░░░░░░░░░████████ 高
      ← 开始 →  ←——— 中间（Lost） ———→  ← 结尾 →
```

**这意味着什么？** 你把 10 个相关 chunk 拼成 context 喂给 LLM，如果答案恰好在第 4-7 个 chunk（中间位置），LLM 可能漏掉它，反而用开头或结尾的不相关信息回答——或者干脆说"找不到"。

**本项目怎么应对？**

没有引入额外的重排策略（如把高相关 chunk 放末尾），但有三层防御：

**① Reranker 精排间接缓解**（`src/pipeline/reranker.ts`）：
```
RRF 融合 25 条 → LLM listwise 精排 → Top-10
```
精排后的 Top-10 比原始 Top-10 相关性高一个档次，意味着即使 LLM 只看开头 3 条也能获得足够信息。

**② 引用验证兜底**（`src/services/chat.service.ts`——`validateAndInjectCitations()`）：
如果 LLM 漏掉了中间某个关键片段导致回答不完整，后处理的 bigram 注入会自动从被遗漏的 source 中补引用来。虽然不能改善回答质量，但至少让用户能点开原文看到遗漏的信息。

**③ 检索量控制**：`RETRIEVAL_TOP_K=25` → `FINAL_TOP_K=10`，确保送入 LLM 的不超过 10 条（约 5000 字），避免上下文过长反而降低精度。

**如果你想更专业地处理**，可以补充这些策略：

| 策略 | 做法 | 代价 |
|------|------|------|
| **重排头尾** | 把最相关的 2 条分别放开头和结尾，次相关的放中间 | 0 额外成本 |
| **两轮检索** | 第一轮粗筛 25 条 → LLM 摘要 → 第二轮精搜 | 2× 生成延迟 |
| **分块问答** | 对超长文档切大段，每段独立回答，再汇总 | N× 生成延迟 |
| **Long-Context LLM** | 用支持 128K+ token 的模型 | API 费用高 |

**面试官喜欢的回答**："我们既认识到了这个问题的存在（展示论文阅读量），也给出了务实的工程方案（Reranker + 引用兜底），不是空谈理论。"

### Q11：长文本场景下，怎么保证回答的前后一致性？

**出现率：65%+ | 难度：偏难**

这个问题的本质是：当 LLM 要基于多个 chunk 的信息生成一段几百字的回答时，怎么保证**第 1 句和第 10 句不互相矛盾**？

**本项目的三层一致性保障**：

**① 引用约束——强制"说话有依据"**（`src/pipeline/generator.ts`——SYSTEM_PROMPT）：

```
规则：
1. 只基于提供的文档内容回答，不要编造任何不在文档中的信息
4. 在回答中使用 [1] [2] 标注引用了哪个参考文档片段
```

引用标注有两个作用：一是用户能验证来源，二是**反向约束 LLM**——要求标注引用意味着 LLM 必须"想清楚"每句话对应哪个文档片段，减少了随口编造的概率。

**② 引用后处理验证**（`src/services/chat.service.ts`——`validateAndInjectCitations()`）：
```typescript
// 剥离无效引用编号（LLM 输出了 [99] 但只有 10 个 source）
answer.replace(/\[(\d{1,2})\]/g, (_match, num) => {
  const n = parseInt(num, 10);
  return n >= 1 && n <= sources.length ? _match : "";
});

// Bigram 兜底注入：LLM 完全没输出引用时自动补
// 对每个句子，计算与每个 source 的 bigram 重叠度，注入最佳匹配
```

**③ system prompt 兜底指令**：
```
5. 使用中文回答
2. 如果文档中没有相关信息，请明确说"根据现有文档，无法回答此问题"
```

这条看似简单的规则其实非常重要——它阻止了 LLM 的"编造冲动"。如果 LLM 被问到一个文档里没有的信息，不写这条规则，它大概率会从训练数据里"借"一段来回答，导致回答与知识库内容矛盾。

**更深层的一致性挑战**（诚实交代当前局限）：

| 场景 | 问题 | 本项目状态 |
|------|------|-----------|
| 跨 chunk 数据矛盾 | 两个 chunk 说了相反的事（旧版 vs 新版文档） | ❌ 未处理——哪个 chunk 排名高信哪个 |
| 数值一致性 | 同一数字在多个 chunk 出现，LLM 可能串版本 | ❌ 依赖文档本身质量（chunk 带时间戳元数据可解） |
| 逻辑链断裂 | 长回答中前半段正确引用文档A，后半段开始自由发挥 | ⚠️ Bigram 兜底部分缓解 |

**面试亮点**："一致性不仅要靠 prompt 约束（让它说真话），还要靠后处理验证（检查它是不是真的说了真话）。信任但验证。"

### Q12：多轮对话的上下文改写怎么做？

### Q14：LLM 提供者切换机制怎么设计的？

## 四-A、上下文窗口与对话管理（新热点 ★★★★★）

### Q15：上下文窗口限制怎么处理？10 个 chunk 塞不下了怎么办？

**出现率：80%+ | 难度：偏难**

这是 RAG 系统的核心约束——LLM 不是无限记忆的，Qwen2.5-7B 上下文窗口约 32K tokens（约 20000+ 中文字）。

**本项目的 token 预算分配**：

```
总窗口：~32K tokens (Qwen2.5-7B)
├── System Prompt： ~200 tokens（5 条规则）
├── 对话历史(6条)： ~2000 tokens（每题 ~150 字 + 回答 ~200 字）
├── context(10 chunks)： ~8000 tokens（每 chunk ~500 字 ≈ 800 tokens × 10）
├── 当前问题： ~200 tokens
└── 输出(max_tokens)： 1024 tokens
──────────────────────────────────────
合计：~11.5K tokens — 占窗口约 36%，留有充足余量
```

**三层防线**：

| 层 | 策略 | 实现位置 |
|----|------|---------|
| **检索量控制** | `RETRIEVAL_TOP_K=25` → `FINAL_TOP_K=10`，硬限制送入 LLM 的 chunk 数 | `config.ts` |
| **内容截断** | BM25 倒排索引只存 chunkId 不存全文本，向量存 LanceDB 文件 | `retrieval.service.ts` |
| **生成截断** | `num_predict=1024`，LLM 回答最长 1024 tokens | `generator.ts` |

**关键设计决策——为什么不用"动态 token 计数截断"？**

更成熟的方案是：拼 context 时逐 chunk 累加 token 数，超过阈值（如 80% 窗口）就停止。本项目当前没做是因为：
1. Qwen2.5 32K 窗口下，10 chunk ≈ 8000 tokens，远未接近上限
2. 项目没有引入 tokenizer 库（如 tiktoken），用字符估算误差大
3. CPU 环境下每次多一次 token 计数 API 调用会增加延迟

**如果要升级**：引入 `tiktoken` 或 `@anthropic-ai/tokenizer`，在 `buildContext()` 中动态截断：

```typescript
// 升级方案（未实现）
function buildContext(sources: ChunkSource[], maxTokens: number): string {
  let tokenCount = 0;
  const selected: ChunkSource[] = [];
  for (const s of sources) {
    const tokens = estimateTokens(s.content);
    if (tokenCount + tokens > maxTokens) break;
    selected.push(s);
    tokenCount += tokens;
  }
  return formatChunks(selected);
}
```

**面试话术**："我们的方案是分层控制——检索层控制候选量（Top-25），精排层控制入参量（Top-10），生成层控制输出量（1024 tokens）。每层各司其职，不需要在一个地方做所有决策。"

### Q16：对话历史长了怎么办？怎么管理多轮对话的内存？

**出现率：70%+ | 难度：中等偏难**

多轮对话的用户体验很好，但技术挑战很大——对话越长，上下文占用越多，留给检索结果的空间越少。

**本项目策略：滑动窗口截断**（`src/services/chat.service.ts` + `generator.ts`——`rewriteQuestion()`）：

```
对话历史管理：
├── 上下文改写：取最近 6 条消息（3 轮对话）
│   └── rewriteQuestion() 中 .slice(-6)
├── LLM 生成：取最近 6 条消息拼接在 system prompt 之后
│   └── getRecentHistory(convId, 6)
└── 完整历史：全量存储在 PostgreSQL messages 表
    └── 用户可以随时滚动查看完整对话
```

**代码验证**（`src/services/chat.service.ts`）：
```typescript
// 每次请求只取最近 6 条
const history = await getRecentHistory(convId, 6);

// 上下文改写：最近 6 条做指代消解
const historyForRewrite = history.map(m => ({ role: m.role, content: m.content }));
const rewrittenQuestion = await rewriteQuestion(question, historyForRewrite);

// 送给 LLM：最近 6 条作为上下文
const historyForLLM = history.map(m => ({ role: m.role, content: m.content }));
```

**为什么是 6 条（3 轮）？**

| 窗口大小 | 优点 | 缺点 |
|---------|------|------|
| 2-4 条 | 省 token | 用户说"第三个方案呢"，前两轮已丢失 |
| 6-8 条 | **平衡点**：覆盖 3 轮上下文 | 更长对话时前面的信息丢失 |
| 全量 | 信息完整 | 10 轮后 context 被历史占满，检索结果塞不下 |

6 条是工程经验值——3 轮对话覆盖了 90% 的指代消解需求（"它""那个""第二个"通常都不超过 3 轮前的指代）。

**对话历史的降级策略**：

| 策略 | 做法 | 本项目 |
|------|------|--------|
| 滑动窗口 | 只保留最近 N 条 | ✅ 当前方案 |
| 摘要压缩 | 对早期对话生成摘要 | ❌ 未实现 |
| 语义过期 | 超过 N 轮的自动标记为"历史上下文" | ❌ 未实现 |
| 向量检索历史 | 用 RAG 的方法检索相关历史消息 | ❌ 未实现 |

**当前局限诚实交代**：用户连续聊 5 轮以上后，第 1 轮的信息会被窗口截断丢掉。如果用户在第 6 轮突然提到第一轮的内容，LLM 无法理解。这是已知的工程 tradeoff——保留全部历史需要更复杂的摘要或检索历史机制。

**面试追问**："怎么改进？"

1. **摘要压缩**：每 4 轮生成一次摘要（"此前讨论了 A、B、C 三个问题，结论是..."），把摘要作为 system prompt 的一部分，释放窗口空间
2. **历史向量检索**：把所有历史消息向量化存入 LanceDB，每轮新问题时同时检索"相关历史消息"，只带回真正相关的历史条目
3. **分层记忆**：短期记忆（最近 3 轮原始消息）+ 中期记忆（4-10 轮摘要）+ 长期记忆（知识库本身）

### Q17：跨 Chunk 信息整合——比如一个问题的答案分散在 3 个不同的 chunk 里，怎么办？

**出现率：60%+ | 难度：偏难**

这是 RAG 系统里常见的"碎片化"问题——假设用户问"性能优化有哪些方法"，方法 A 在 chunk 3，方法 B 在 chunk 7，方法 C 在 chunk 9。

**本项目的处理方式**：

检索层已经把 3 个 chunk 都召回了（RRF 融合 + Reranker 精排），LLM 在 `generate()` 阶段拿到的是：

```
参考文档：
[1] 性能优化指南.pdf
方法 A：使用 Redis 缓存热点数据...（chunk 3）

[2] 性能优化指南.pdf
方法 B：数据库索引优化...（chunk 7）

[3] 性能优化指南.pdf
方法 C：前端懒加载和代码分割...（chunk 9）

问题：性能优化有哪些方法？
```

LLM 会自然地整合 3 个 chunk 的信息输出答案。这是 LLM 的内建能力——信息整合，不需要额外编程。

**但有两个工程挑战**：

**挑战 1：排名偏差**——如果 Reranker 把方法 B 排在 chunk 1，方法 A 排在 chunk 10，Lost in the Middle 效应可能导致 LLM 忽略方法 A。

→ 本项目用 Reranker 缓解，确保 Top-3 的 chunk 覆盖了最相关的信息。

**挑战 2：重复/矛盾信息**——同一个信息在多个 chunk 中出现（比如方法 A 在 chunk 3 和 chunk 8 都被提到，但描述略有不同），LLM 可能产生混乱。

→ 本项目依赖文档本身的质量，没有做 chunk 去重（当前数据规模下不是瓶颈）。

**如果要做得更专业**：

| 方法 | 做法 | 代价 |
|------|------|------|
| **Map-Reduce** | 对每个 chunk 独立提问 → 汇总所有回答 → LLM 再次总结 | 3× LLM 调用 |
| **Refine** | 逐 chunk 迭代精炼："结合 chunk 1，还有什么补充"→ chunk 2... | N× LLM 调用 |
| **多轮检索** | 第一轮找到相关 chunk → 用第一轮结果扩大搜索范围 → 第二轮精搜 | 2× Embedding |

**面试话术**："当前场景下 10 chunk × 500 字 = 5000 字足够覆盖大多数问题。跨 chunk 整合是 LLM 的天然能力，我们不额外干预。但如果遇到特别复杂的分析型问题，Map-Reduce 是下一步方案。"

## 五、后端架构（高频 ★★★★）

### Q18：JWT 双 Token 认证怎么设计的？refresh rotation 机制？

**出现率：80%+ | 难度：中等**

```
access token（短）：15 分钟过期，内存保存
refresh token（长）：7 天过期，DB 存储 + localStorage
```

**Refresh Rotation 机制**（`src/services/auth.service.ts`）：
1. 用户用旧 refresh token 请求 `/api/auth/refresh`
2. 服务端验证旧 token → **删除旧 token → 签发新 token 对**
3. 如果旧 token 已被使用过（已删除），说明 token 可能被盗用 → 删除该用户全部 refresh token → 强制重新登录

**前端无感刷新**（`client/src/lib/api.ts`）：
```typescript
// 401 拦截 → 尝试 refresh → 用新 access token 重试原请求
// refreshPromise 单例：并发请求同时 401 时只发一次 refresh
if (!refreshPromise) {
  refreshPromise = fetch("/api/auth/refresh", { ... }).finally(() => {
    refreshPromise = null;
  });
}
await refreshPromise;
```

**API Key 鉴权（V11）**：`ak_` 前缀，HMAC-SHA256 哈希存储，明文仅创建时返回一次。Chat 路由同时支持 JWT 和 API Key（OR 逻辑，`authenticateApiKey` 中间件）。

### Q19：多租户数据隔离怎么做的？

**出现率：70%+ | 难度：中等**

**方案**（`src/middleware/auth.ts`——`requireKBAccess`）：

```sql
-- kb_members 表（V4）
kb_id   | user_id  | role    | created_at
--------|----------|---------|-----------
```

### Q20：错误处理怎么设计的？

## 六、系统设计与优化（高频 ★★★★★）

### Q21：答案缓存怎么设计？缓存失效策略？

**出现率：75%+ | 难度：中等偏难**

**本项目方案**（`src/lib/cache.ts`——V14）：

```
缓存 Key：cache:{kbId}:{model}:{sha256(归一化问题)}
TTL：24 小时（Redis SETEX）
失效策略：文档上传/删除/重试时 → invalidateKB(kbId) 清除该 KB 全部缓存
```

**为什么用 SHA256 hash 而不是直接用问题文本？**
1. Redis key 长度限制——问题可能上 100 字，hash 取前 16 位就够了
2. 归一化处理（去空格 + 统一小写）→ 同样意思不同写法的问题命中同一缓存

**为什么是全部清除（invalidateKB）而不是精确清除？** 文档变更后，可能影响多个问题的检索结果（因为向量空间变了），无法精确知道哪些缓存条目受影响。全部清除是 conservative 策略——宁可多失效一些，也不留脏缓存数据。

**命中率统计**：内存计数器（不是 Redis 的 INFO 命令），进/出缓存时 `recordHit()`/`recordMiss()`。管理后台有"性能"标签查看。

### Q22：向量迁移怎么做的？为什么需要它？

**出现率：60%+ | 难度：偏难**

**场景**：V13 引入了 Embedding 模型切换能力——用户在管理后台从 BGE-m3 换成其他模型后，所有已索引的向量是用 BGE-m3 生成的，新模型生成的向量维度/数值分布不同 → 需要全部重建索引。

**本项目方案**（`src/lib/vector-migration.ts`）：

```
1. 创建新 LanceDB 表（v2 → v3 版本递增）
2. 读取所有 chunk，用新 Embedding 模型重新生成向量
3. 写入新表
4. 更新 knowledge_bases.vector_version → 切换到新表
5. 成功 → 删除旧表；失败 → 回滚到旧版本
```

**迁移进度追踪**：`knowledge_bases.migration_status` 字段 + `GET /api/admin/ai-engine/migrate/:kbId/progress` 端点。

**面试考点**：这个设计的巧妙之处在于**不回滚数据**——旧表保留到迁移成功确认后才删除，即使迁移过程中服务重启也不会丢数据。

## 九、踩坑与行为面试（必问 ★★★★★）

### Q26：这个项目你遇到的最大技术挑战是什么？

**出现率：95%+ | 难度：行为面试**

**回答框架（STAR 法则）**：

**Situation**：项目是 CPU Only 环境（Intel 核显，无 GPU），LLM 和 Embedding 都跑在 Ollama CPU 上，延迟很高。

**Task**：需要在不增加硬件的条件下优化检索到回答的端到端延迟。

**Action**：
1. **Reranker 优化（V14）**：原方案用 similarityRerank 对 25 条结果逐条计算相似度，需要 25 次 Embedding API 调用。改成 LLM listwise 方案后，一次 API 调用完成精排，延迟从 25s 降到 3s。
2. **答案缓存（V14）**：Redis 缓存同问题 24h 内重复提问直接返回，命中时延迟从 15-30s 降到 <100ms。
3. **流式输出（V2）**：用户不需要等全部生成完，首 token 渲染时间 <2s。
4. **增量索引（V3）**：文档更新时只处理变化的部分，避免全量重建。

**Result**：端到端延迟从 V1 的 20-40s 优化到 V14 的 <5s（缓存命中时 <100ms）。用户可感知的等待时间从"能泡杯咖啡"变成"来得及眨眼"。

**面试官喜欢的点**：展示了"在没有更好硬件的情况下，通过架构优化解决问题"的能力。这是一个很真实的场景——大多数公司不是 OpenAI，没有几千张 A100。

### Q27：如果我要把系统扩展到支持 100 万文档，你会改什么？

**出现率：70%+ | 难度：系统设计**

1. **LanceDB → Milvus/Qdrant**：LanceDB 适合中小规模，百万级别需要分布式向量库。Milvus 支持分片、副本、混合查询。

## 十、Agent 实战亮点与深度工程实践（高区分度 ★★★★★）

> 以下 6 道题是真正区分"用过 RAG"和"做深了 RAG"的分水岭。面试官听到这些细节会知道你在项目中做了真正的工程决策，而不是调了个 API。

### Q29：Agent 函数调用是怎么嵌入 RAG 流程的？为什么放在检索前面而不是后面？

**出现率：70%+ | 难度：偏难 | 区分度：极高**

**核心设计决策**：Agent 拦截发生在检索**之前**，不是之后。

```
用户提问 → Agent检测 ──命中(时间/计算器)──→ 直接返回函数结果（跳过检索+生成）
                        └──未命中──→ rewriteQuestion → retrieve → generate
```

**为什么不在检索之后？** 因为"现在几点""1+2×3 等于多少"这类问题不需要检索知识库。先检索再拦截 = 白白做了一次 Embedding + 向量搜索 + BM25 + RRF 融合，浪费了 CPU 资源。

**本项目实现**（`src/pipeline/agent.ts`）：

```typescript
// 1. 函数注册表模式
const functions: FunctionDef[] = [
  { name: "get_current_time", handler: async () => new Date().toLocaleString("zh-CN") },
  { name: "simple_calculator", handler: async (args) => { /* Function constructor 沙箱 */ } },
];

// 2. LLM 意图检测（temperature=0 保证确定性）
const prompt = `判断用户问题是否需要调用函数。只返回 JSON。
用户问题：${question}`;

// 3. JSON 解析 + 函数路由 + 降级
const json = JSON.parse(data.message.content);
const fn = functions.find(f => f.name === json.function);
if (!fn) return { isFunctionCall: false, result: null };  // 不认识的函数，放行给检索
const result = await fn.handler(json.args);
return { isFunctionCall: true, result };
```

**三个工程细节面试官会追问**：

**① 为什么 temperature=0？** 意图检测需要确定性——同一个问题每次必须给出相同的判断结果。如果用 temperature=0.7，"现在几点"可能有时被识别为函数调用、有时被当做普通对话，用户体验很糟糕。

**② LLM 返回非 JSON 怎么办？** JSON 解析在 try-catch 里，异常直接返回 `{ isFunctionCall: false }`，放行给检索流程。不会因为 Agent 解析失败导致整个问答不可用。

**③ simple_calculator 的安全沙箱？**
```typescript
// 正则白名单过滤 + Function constructor 沙箱
const expr = String(args.expression).replace(/[^0-9+\-*/().%\s]/g, "");
const result = Function(`"use strict"; return (${expr})`)();
```
`Function constructor` + `"use strict"` 创建隔离作用域，无法访问 `require`/`process`/全局变量。正则过滤只允许数学运算字符。虽然不如 Docker 沙箱安全，但对内部系统足够。

**架构意义**：Agent 在 RAG 流程中的位置体现了"先判断、后计算"的工程思维——不是所有问题都需要走完整的 RAG 管道。功能开关 `AGENT_ENABLED=true/false` 允许生产环境动态开关。当 Agent 功能稳定后再打开，不稳时就关掉，不阻塞核心问答。

### Q30：增量索引的 Hash 比对算法具体怎么做的？怎么判断哪些 chunk 要删除、哪些要保留？

**出现率：55%+ | 难度：偏难 | 区分度：极高**

这是一道**算法题级别**的面试题。核心问题是：用户重新上传同名文档后，新旧 chunk 如何做最小化更新？

**本项目方案**（`src/services/doc.service.ts`——`processDocumentAsync()`）：

```
三集合比对算法：

oldHashes = {h1→chunkId_1, h2→chunkId_2, h3→chunkId_3}  // 旧 chunk 的 hash 映射
newHashes = {h2, h3, h4}                                    // 新 chunk 的 hash 集合

              oldHashes
            ┌───────────┐
            │  h1       │  ← 待删除（旧有，新无）
            │           │
    ┌───────┤  h2  h3  ├───────┐
    │       │           │       │
    │       └───────────┘       │
    │                           │
    ▼                           ▼
  复用 h2, h3           newHashes
  (更新 PG 指向新 docId)         │
                                h4  ← 需要重新 embedding
```

**代码实现**：
```typescript
// 比对逻辑
const oldHashSet = new Map<string, string>(); // hash → oldChunkId
for (const oc of oldChunks) {
  const h = oc.metadata?.contentHash;
  if (h) oldHashSet.set(h, oc.id);
}

const newHashes = new Set(textChunks.map(c => c.contentHash));

// 1. 复用：hash 同时存在于新旧集合 → 更新 PG 指向新 docId
for (const [hash, chunkId] of oldHashSet) {
  if (newHashes.has(hash)) {
    await db.update(chunks).set({ docId, chunkIndex: ... })
      .where(eq(chunks.id, chunkId));
  }
}

// 2. 删除：旧 hash 不在新集合中 → 从 LanceDB 清向量
const staleIds = [];
for (const oc of oldChunks) {
  if (!newHashes.has(oc.metadata?.contentHash)) {
    staleIds.push(`${docId}_${oc.chunkIndex}`);
  }
}
await deleteVectors(kbId, staleIds);

// 3. 新增：新 hash 不在旧集合中 → 只对这些做 embedding
chunksToEmbed = textChunks.filter(c => !oldHashSet.has(c.contentHash));
```

**三个关键决策**：

**① 只用 SHA256 hash，不做语义去重**。语义去重需要额外 Embedding 调用（"这篇文档今天重新上传，虽然文字稍有改动但意思一样"→ 需要比较向量相似度）。当前用精确比对是更保守的方案——宁可多做 embedding 也不少做。

**② 复用 chunk 直接更新 PG，不重写 LanceDB**。LanceDB 是 Append-Only 的结构，频繁删改代价大。所以复用 chunk 只需更新 PG 的 docId 指针，向量本身不动。只有 hash 对不上的才删旧写新。

**③ 功能开关**：`INCREMENTAL_INDEX=false` 时关闭所有比对逻辑，走全量重建。新功能默认关闭，生产验证后再打开——渐进式交付。

**面试话术**："增量索引看起来是功能优化，本质是一个集合运算问题。三集合比对（新增/复用/删除）用 O(n) 的 hash 查找就能解决，不需要对每个 chunk 做 embedding 去查向量相似度。"

### Q31：全局降级策略是怎么设计的？为什么每个模块都有独立的降级而不是一个统一的 try-catch？

**出现率：60%+ | 难度：中等偏难 | 区分度：高**

这是最能体现"系统工程思维"的问题。看看项目中有多少降级逻辑：

| 模块 | 主方案 | 降级方案 | 降级触发条件 |
|------|--------|---------|------------|
| **中文分词** | `@node-rs/jieba` (Rust 版本) | 字符 bigram + 词级切分 | jieba 加载失败（import 异常） |
| **Reranker** | `llmListwiseRerank()` | 返回原始 Top-10 候选 | LLM 调用失败或超时 |
| **LLM 提供者** | DeepSeek API（云端，秒回） | Ollama CPU（本地，慢但免费） | API 调用失败或 `DEEPSEEK_API_KEY` 未配置 |
| **BM25 检索** | Redis `smembers` | 跳过该 term，继续其他 term | Redis 连接失败 |
| **Redis 缓存读** | 返回缓存答案 | `return null`，走正常检索 | Redis `get` 异常 |
| **Redis 缓存写** | 写入缓存 | `.catch(() => {})`，静默跳过 | Redis `setex` 异常 |
| **向量删除** | `deleteVectors()` | 跳过，不阻塞 PG 操作 | LanceDB 写入/删除异常 |
| **Agent 检测** | JSON 解析 + 函数调用 | 返回 `{ isFunctionCall: false }` 放行检索 | JSON 解析失败、函数名不匹配 |
| **上下文改写** | LLM 改写问题 | 返回原始问题 | LLM 调用异常 |
| **引用注入** | `validateAndInjectCitations()` | bigram 兜底注入 | LLM 未输出任何引用编号 |

**设计原则**：

```
铁律：任何非核心模块的失败都不能阻断核心流程（问答）。
核心流程 = retrieve() + generate()
非核心 = Agent检测、Reranker、缓存、BM25单个term
```

**为什么不做一个全局 try-catch？** 全局 try-catch 的问题是：你不知道"哪一层"出了问题，降级粒度太粗。

```
// 不好的做法：全局兜底
try {
  result = await fullPipeline();
} catch {
  result = fallback();
}
// 问题：分词失败 → 降级了整个检索？不合理，应该只降级分词
```

**本项目的做法：每层独立降级**：
```typescript
// 好的做法：分层降级
// 分词层
try { cutFn = jieba } catch { cutFn = bigram; }

// Reranker层
try { sources = await rerank(); } catch { /* 返回原始候选 */ }

// 缓存层
try { cached = await getCached(); } catch { cached = null; }
```

**面试金句**："降级不是'出错了怎么办'，是'这一层出错了，上一层和下一层继续怎么办'。每层的降级策略要保证对相邻层的影响最小化。"

### Q32：前端并发 refresh token 去重 + accessToken 内存安全的双重设计是怎么做的？

**出现率：55%+ | 难度：中等 | 区分度：高**

这是一个前端安全面试的经典题，本项目有两个精妙设计。

**设计一：refreshPromise 并发去重**

**问题场景**：页面加载时同时发出 5 个 API 请求（KB 列表、对话列表、用户信息、文档列表、统计数据），全部 401。如果不做并发去重 → 5 个请求各自触发 refresh → 5 次 refresh token rotation → 前 4 个 refresh token 依次被"用旧发新"吊销 → 第 5 个请求拿到了已被吊销的 refresh token → 强制登出。

**解决方案**（`client/src/lib/api.ts`——`refreshAccessToken()`）：

```typescript
let refreshPromise: Promise<string | null> | null = null;

async function refreshAccessToken(): Promise<string | null> {
  if (refreshPromise) return refreshPromise;  // ← 关键：单例模式

  refreshPromise = (async () => {
    try {
      const res = await fetch("/api/auth/refresh", { ... });
      // 用旧 token 换新 token → 服务端 rotation
      accessToken = data.accessToken;
      setRefreshToken(data.refreshToken);
      return accessToken;
    } finally {
      refreshPromise = null;  // ← 释放锁
    }
  })();

  return refreshPromise;
}
```

5 个请求同时 401 → 第一个进入时 `refreshPromise` 为 null → 创建 refresh 请求 → 后 4 个看到 `refreshPromise` 非 null → 全部 await 同一个 promise → 只发一次 refresh → 5 个请求都用新 token 重试。

**面试追问**："如果 refresh 请求本身也失败了怎么办？"
→ `clearAuth()` + 跳转登录页。所有正在等待的请求会跟着拿到 null → 各自的 401 处理 → 统一跳转。不会出现部分请求拿到新 token、部分拿不到的不一致状态。

**设计二：accessToken 内存 vs refreshToken localStorage 的安全分层**

```typescript
let accessToken: string | null = null;           // 只在内存中（页面刷新清空）
localStorage.setItem("refreshToken", token);     // localStorage（持久化）

// 安全原理：
// accessToken 在内存 → XSS 攻击拿不到（只有当前标签页的 JS 能访问闭包变量）
// refreshToken 在 localStorage → XSS 能拿到，但有 rotation 机制（用旧发新，旧即失效）
```

| 攻击场景 | accessToken (内存) | refreshToken (localStorage) |
|---------|-------------------|---------------------------|
| XSS 读 localStorage | ❌ 读不到 | ✅ 能读到 |
| 页面刷新 | 丢失（需要重新 refresh） | 保留 |
| CSRF | 不受影响（SameSite cookie x） | 不受影响 |
| XSS 拿到 refreshToken | — | rotation 机制限制攻击窗口 |

**面试金句**："accessToken 放内存是为了防止 XSS 一次性偷走长期凭证，refreshToken 放 localStorage 是为了页面刷新不用重新登录——两者各有代价，rotation 机制补上了 localStorage 的安全短板。"

### Q33：文档处理的 progressStep 五步状态机怎么设计的？为什么需要这么细的粒度？

**出现率：50%+ | 难度：中等 | 区分度：中高**

**问题背景**：文档上传后是异步处理的（用户上传即返回，后台慢慢处理）。在 CPU 环境下，一个 50 页 PDF 可能需要 1-2 分钟。用户看到"处理中"，等了 1 分钟还不知道进度，体验很差。

**本项目方案**（`src/services/doc.service.ts`——`processDocumentAsync()`）：

```
progressStep 状态机：
parsing → splitting → embedding → indexing → ready
   ↓         ↓           ↓           ↓          ↓
 解析PDF    切分段落    向量化       BM25+PG    完成
```

```typescript
// 每一步切换时写入 progressStep
await db.update(documents).set({ progressStep: "parsing" })   // 解析中
// ... parseFile ...
await db.update(documents).set({ progressStep: "splitting" })  // 切分中
// ... splitText ...
await db.update(documents).set({ progressStep: "embedding" })  // 向量化中
// ... embed ...
await db.update(documents).set({ progressStep: "indexing" })   // 索引入库中
// ... insertVectors + insertChunks + indexBM25 ...
await db.update(documents).set({ status: "ready", progressStep: "ready" })  // 完成

// 异常时
await db.update(documents).set({ status: "error", errorMessage, progressStep: null })
```

**前端展示**（`DocList.vue`）：根据 `progressStep` 显示不同阶段文字 —— "正在解析..." / "正在切分..." / "正在向量化..." / "正在入库..."。

**两个工程细节**：

**① 为什么状态是"先写 DB 再执行"而不是"执行完再写"？**

```
// 本项目做法：先写状态，再执行
await db.update().set({ progressStep: "embedding" });  // 写 DB
const vectors = await embed(texts);                     // 执行

// 为什么？因为执行可能超时（CPU 下 embed 很慢）
// 如果执行完再写 → 执行期间 status 一直是 "splitting"
// → 前端轮询会一直显示"切分中"，用户以为卡了
```

**② 失败文档汇聚 + 重试**（V12 的 `AdminErrorDocs.vue`）：

```
管理后台"失败文档"标签：
  ├── 跨 KB 汇聚所有 status=error 的文档
  ├── 显示文件名、所属 KB、错误原因、上传时间
  └── 一键重试：POST /api/kb/:id/docs/:docId/retry
       → 重置 status=processing → 重新走 progressStep 状态机
```

**面试话术**："文档处理的进度追踪是用户体验和运维效率的交叉点。普通用户看到进度条不焦虑，管理员看到失败文档能一键重试而不需要进数据库修数据。这就是可观测性在业务层的落地。"

### Q34：LanceDB 版本切换实现向量迁移零中断是怎么设计的？

**出现率：45%+ | 难度：偏难 | 区分度：极高**

**问题场景**（V13）：用户从 BGE-m3 切换 Embedding 模型后，所有已有向量是用旧模型生成的 → 新查询的向量（新模型生成）和旧索引的向量不在同一个向量空间 → 检索精度归零。

**本项目方案**（`src/lib/vector-migration.ts`）：

```
迁移流程（零中断）：
1. 用户访问管理后台 → 选新 Embedding 模型 → 点击"开始迁移"
2. 后端：创建新 LanceDB 表（表名带版本号如 vectors_v3）
3. 后端：逐个 chunk 用新模型重新 embed → 写入新表
   ├── 同时：旧表 vectors_v2 继续提供服务（问答正常）
   └── progressStep 追踪：0→1(chunks loaded)→2(embedding)→3(writing)→4(switching)→5(complete)
4. 新表写入完成 → knowledge_bases.vector_version: "v3"
5. 检索时：searchVectors(kbId) 读取 vector_version → 查新表
6. 成功确认 → 删除旧表 vectors_v2
   失败 → 回滚 vector_version 到 "v2"，删除残留新表
```

**为什么这个设计是"零中断"？**

```
时间线：
t0: 用户问答 → 查 vectors_v2（旧表，正常服务）
t1: 迁移开始 → 写 vectors_v3（新表，不影响查询）
t2: 迁移进行 → 问答查 vectors_v2（旧表，正常服务）
t3: 迁移完成 → vector_version 切换到 v3 → 问答查 vectors_v3（新表，正常服务）
t4: 清理旧表
```

关键：**迁移过程中旧表始终在线**。迁移是一个"影子写入"操作——新表默默构建，完成后再原子切换。

**面试追问**："如果切换后用户说新模型不好用怎么办？"
→ `rollbackMigration()` 将 `vector_version` 切回旧版本号，删除新表。因为旧表在清理前一直保留，回滚是瞬间完成的。

**面试金句**："数据迁移的第一原则是'可回滚'。宁可迁移后多保留一会儿旧数据占磁盘，也不能在确认成功前删旧数据。磁盘空间是便宜的，数据丢失是昂贵的。"
