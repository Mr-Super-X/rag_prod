## 1. 系统概览

### 1.1 产品定位

面向 5-20 人技术团队的内部知识库问答平台。核心价值：上传团队文档 → 自然语言提问 → 基于文档内容精确回答 + 引用来源。

```
│  Vue 3 SPA  →  登录/注册  →  知识库管理  →  问答交互            │
│  localhost:5173          Vite dev server + 代理到 3000         │
└────────────────────────────┬─────────────────────────────────┘
```

### 1.3 关键设计决策

| 决策 | 选择 | 替代方案 | 为什么 |
|------|------|---------|--------|
| 向量库嵌入进程 | LanceDB | Qdrant (Docker) | 国内 Docker 镜像源不稳定，嵌入方案零外部依赖 |
| LLM + Embedding 统一管理 | Ollama | Transformers.js + Ollama | 减少模型服务运行时，降低运维复杂度 |
| 异步文档处理 | Fire-and-forget | 消息队列 | 当前用户规模不需要复杂队列 |
| 前端状态管理 | Pinia | Vuex | Vue 3 官方推荐，更轻量 |
| ORM | Drizzle | Prisma | SQL-like 语法直观，bundle 小，migration 自动生成 |
| 认证方案 | JWT access(15m) + refresh(7d rotation) | Session/Cookie | 无状态扩展方便，refresh rotation 安全性好 |
| LLM 提供者 | Ollama 本地 + DeepSeek 云端降级 | 单一提供者 | 本地免费但慢，云端快但需出网，混合兼顾 |

## 2. 数据模型

### 2.1 实体关系图

```
users (用户)
  │
  ├── 1:N ── knowledge_bases (知识库)
  │             │
  │             ├── 1:N ── kb_members (知识库成员)
  │             │             └── userId + kbId + role
  │             │
  │             ├── 1:N ── documents (文档)
  │             │             │
  │             │             └── 1:N ── chunks (文本块)
  │             │                         │
  │             │                         └── 1:1 ── LanceDB vectors (向量)
  │             │
  │             └── 1:N ── conversations (对话)
  │                           │
  │                           └── 1:N ── messages (消息)
  │                                       │
  │                                       ├── JSONB sources (引用来源)
  │                                       │
  │                                       └── 1:N ── message_feedback (反馈)
  │
  ├── 1:N ── refresh_tokens (刷新令牌)
  ├── 1:N ── api_keys (API 密钥，HMAC-SHA256 哈希存储)
  └── 1:N ── audit_logs (操作审计)
```

### 2.2 表结构详解

#### users

```
id          UUID    PRIMARY KEY    用户唯一标识
username    VARCHAR(50) UNIQUE    登录用户名
passwordHash VARCHAR(255)         bcrypt 哈希（salt rounds=10）
role        VARCHAR(20)           'admin' | 'user'
createdAt   TIMESTAMPTZ           注册时间
```

**设计说明**：
- 用 UUID 而非自增 ID —— 防止用户枚举，分布式友好
- `role` 存字符串而非外键表 —— 两种角色够用，不引入额外 JOIN

#### knowledge_bases

```
id          UUID    PRIMARY KEY
name        VARCHAR(200)         知识库名称
description TEXT                  可选描述
createdBy   UUID    → users.id   创建者
createdAt   TIMESTAMPTZ
updatedAt   TIMESTAMPTZ
```

**设计说明**：
- `createdBy` 字段实现"我的知识库"标记
- KB 访问通过 `kb_members` 表控制：创建者自动为 owner，其他用户需被添加为 member

#### documents

```
id          UUID    PRIMARY KEY
kbId        UUID    → kbs.id (CASCADE)  所属知识库
filename    VARCHAR(500)                原始文件名
fileType    VARCHAR(20)                 pdf|docx|xlsx|pptx|md|txt
fileSize    BIGINT                      字节数
storagePath VARCHAR(500)               磁盘存储路径
status      VARCHAR(20)                 queued|processing|ready|error
chunkCount  INT                        切分后的文本块数
uploadedBy  UUID    → users.id         上传者
errorMessage TEXT                      失败原因
createdAt   TIMESTAMPTZ
updatedAt   TIMESTAMPTZ                最后状态变更时间
```

**设计说明**：
- `status` 状态机：queued → processing → ready | error。前端轮询展示处理进度
- `storagePath` 保存文件物理路径 —— Docker volume 持久化
- `ON DELETE CASCADE` —— 删 KB 自动清理所有文档

#### chunks

```
id              UUID    PRIMARY KEY
docId           UUID    → documents.id (CASCADE)
kbId            UUID    → kbs.id (CASCADE)    冗余字段，避免跨表 JOIN
qdrantPointId   VARCHAR(100)                  对应 LanceDB 向量 ID
content         TEXT                          文本块内容
chunkIndex      INT                           在文档中的序号
metadata        JSONB                         扩展元数据
createdAt       TIMESTAMPTZ
```

**设计说明**：
- `kbId` 故意冗余 —— 检索时按 KB 查 chunks 不需要 JOIN documents，减少一次查询
- `qdrantPointId` 字段名保留（最初选的 Qdrant，后改为 LanceDB）—— 对 PG 无影响，重命名涉及 migration，暂不改

#### conversations

```
id          UUID    PRIMARY KEY
kbId        UUID    → kbs.id (CASCADE)
userId      UUID    → users.id
title       VARCHAR(300)                自动生成（首条消息截取）
createdAt   TIMESTAMPTZ
updatedAt   TIMESTAMPTZ
```

**设计说明**：
- `updatedAt` 用于排序"最近活跃"的对话
- `title` 初始为"新对话"，首轮问答后自动截取用户问题前 50 字

#### messages

```
id              UUID    PRIMARY KEY
conversationId  UUID    → conversations.id (CASCADE)
role            VARCHAR(20)             'user' | 'assistant'
content         TEXT                    消息内容
sources         JSONB                   引用来源数组
createdAt       TIMESTAMPTZ
updatedAt       TIMESTAMPTZ
```

**sources JSONB 结构**：
```json
[
  {
    "chunkId": "doc123_0",
    "content": "引用原文片段",
    "score": 0.032,
    "docFilename": "考勤制度.md"
  }
]
```

#### kb_members

```
id          UUID    PRIMARY KEY
kbId        UUID    → kbs.id (CASCADE)
userId      UUID    → users.id (CASCADE)
role        VARCHAR(20)             'owner' | 'member'
createdAt   TIMESTAMPTZ
```

**设计说明**：
- KB 多租户隔离的核心表。创建者写入 `owner` 角色，被添加的成员为 `member`
- 删除 KB 时级联清除所有成员记录

#### refresh_tokens

```
id          UUID    PRIMARY KEY
userId      UUID    → users.id (CASCADE)
token       VARCHAR(500) UNIQUE    随机 96 位 hex
expiresAt   TIMESTAMPTZ            创建时 +7 天
createdAt   TIMESTAMPTZ
```

**设计说明**：
- refresh token rotation：每次 refresh 用旧 token 换新 token，旧 token 立即删除
- access token 15 分钟内存，refresh token 7 天 localStorage，401 时自动静默刷新

#### message_feedback

```
id          UUID    PRIMARY KEY
messageId   UUID    → messages.id (CASCADE)
userId      UUID    → users.id (CASCADE)
rating      INT                     1=赞, -1=踩
createdAt   TIMESTAMPTZ
updatedAt   TIMESTAMPTZ
```

**设计说明**：
- 同一用户同一消息唯一评价，再次点击同一按钮取消，切换按钮更新记录

#### audit_logs

```
id          UUID    PRIMARY KEY
userId      UUID    → users.id (CASCADE)
action      VARCHAR(50)             login|create_kb|delete_kb|upload_doc|delete_doc
resource    VARCHAR(50)             kb|document|user
resourceId  VARCHAR(100)
ip          VARCHAR(45)
createdAt   TIMESTAMPTZ
```

**设计说明**：
- fire-and-forget 记录，不阻塞业务流程
- admin 可在管理控制台查看和筛选

#### api_keys

```
id          UUID    PRIMARY KEY
userId      UUID    → users.id (CASCADE)
keyHash     VARCHAR(255) UNIQUE    HMAC-SHA256(JWT_SECRET, plaintext_key)
keyPrefix   VARCHAR(8)             明文前 8 位，用于 UI 展示
name        VARCHAR(100)           用户自定义名称
lastUsedAt  TIMESTAMPTZ
createdAt   TIMESTAMPTZ
revokedAt   TIMESTAMPTZ            NULL=有效，非 NULL=已吊销
```

**设计说明**：
- 明文密钥仅创建时一次性返回（`ak_` 前缀 + 随机字符串），数据库中仅存哈希
- 吊销操作不物理删除，`revokedAt` 设当前时间
- 问答 API 支持 `Authorization: Bearer ak_xxx` 鉴权

### 2.3 LanceDB 向量存储

与 PostgreSQL 的 chunks 表**并行存在**。两者的对应关系：

| PostgreSQL chunks | LanceDB kb_{kbId} |
|-------------------|-------------------|
| `qdrantPointId` | `id` 字段 |
| `content` | `content` 字段 |
| — | `vector` 字段（1024 维浮点数组） |
| `chunkIndex` | `chunkIndex` 字段 |

**不合并的原因**：
- 向量数据不适合存关系数据库（pgvector 千万级以上性能下降）
- LanceDB 专为向量搜索优化（HNSW 索引），比 PG 的 IVFFlat 快一个数量级

## 3. 核心链路详解

### 3.1 文档摄入全链路

```
用户上传文件
  │
  ▼
POST /api/kb/:id/docs (multipart/form-data)
  │
  ├─ Fastify @fastify/multipart 解析
  ├─ 文件扩展名校验 (.pdf|.docx|.xlsx|.pptx|.md|.txt)
  ├─ 保存到 ./data/uploads/{timestamp}_{filename}
  │
  ▼
processDocument()  ← 立即返回 docId，不阻塞
  │
  ├─ INSERT INTO documents (status="processing")
  │
  └─ processDocumentAsync()  ← 后台执行
       │
       ├─ parseFile()
       │   ├─ .txt/.md → fs.readFile (utf-8)
       │   ├─ .pdf     → pdf-parse (提取文本)
       │   └─ .docx    → mammoth (提取文本)
       │
       ├─ splitText()
       │   ├─ 按段落分割 (\n\s*\n)
       │   ├─ chunk_size=500 字, overlap=50 字
       │   └─ 返回 TextChunk[]
       │
       ├─ embed()
       │   ├─ 每批 5 个 chunk 调用 Ollama /api/embed
       │   ├─ 模型: bge-m3, 输出: 1024 维向量
       │   └─ 超时 5 分钟/批
       │
       ├─ insertVectors()
       │   ├─ LanceDB: 首次创建表, 已存在则追加
       │   ├─ 每条数据: { id, vector, chunkId, docId, kbId, content, chunkIndex, docFilename }
       │   └─ 向量列自动识别为 FixedSizeList(1024)
       │
       ├─ INSERT INTO chunks (逐条)
       │
       ├─ indexBM25()
       │   ├─ @node-rs/jieba 中文分词（失败降级为字符 bigram）
       │   ├─ Redis SADD bm25:{kbId}:{term} {chunkId}
       │   └─ 多词用 Pipeline 批量执行
       │
       └─ UPDATE documents SET status="ready", chunkCount=N
```

**为什么异步处理**：
- CPU 模式下，50KB 文档的 Embedding 需要 1-2 分钟
- 同步处理会让 HTTP 连接挂起直到超时
- 异步方案：响应秒回（仅 INSERT），后台静默处理

### 3.2 检索全链路

```
用户提问: "年假有多少天？"
  │
  ▼
POST /api/kb/:id/chat
  │
  ├─ chat.service.ask()
  │
  ├─ 1. 上下文改写
  │   ├─ 取最近 6 条历史消息
  │   ├─ 调 Ollama /api/chat（temperature=0.1, max_tokens=100）
  │   ├─ 输入: 历史对话 + "那婚假呢？"
  │   ├─ 输出: "公司的婚假政策是什么？"
  │   └─ 超时 2s → 退回原始问题
  │
  ├─ 2. 向量检索 (LanceDB)
  │   ├─ embedSingle(改写后的问题) → 1024 维向量
  │   ├─ LanceDB table.search(vector).limit(25)
  │   └─ 返回 Top-25 + 余弦距离
  │
  ├─ 3. 关键词检索 (BM25)
  │   ├─ @node-rs/jieba 分词 → ["年假", "多少天"]
  │   ├─ 每个词查 Redis: SMEMBERS bm25:{kbId}:{term}
  │   ├─ 统计每个 chunkId 命中多少词
  │   └─ 返回 Top-25 + 命中率分数
  │
  ├─ 4. RRF 融合 (Reciprocal Rank Fusion)
  │   ├─ RRF_score(d) = Σ 1/(k + rank_r(d))
  │   ├─ k = 60（平滑参数）
  │   ├─ 向量排名 + BM25 排名 → 综合分数
  │   └─ 返回 Top-10
  │
  ├─ 5. 兜底判断
  │   └─ sources.length === 0 → "未找到相关内容"
  │
  └─ 6. LLM 生成
      ├─ System Prompt: "基于文档回答，不要编造"
      ├─ User: 检索到的 chunk 内容 + 问题
      ├─ Ollama /api/chat (temperature=0.3, num_predict=1024)
      └─ 返回 { answer, sources }
```

#### 3.2.1 Reranker 精排方案演变

Reranker 在 RRF 融合后对候选结果做精排，从 Top-25 选出 Top-5。经历了两版方案。

**V1 方案（逐条打分，已废弃）**

```
候选1 → LLM("给0-1分数") → 0.83
候选2 → LLM("给0-1分数") → 0.51
...
候选25 → LLM("给0-1分数") → 0.67
        ↓
    按分数排序 → Top-5
```

| 优点 | 缺点 |
|------|------|
| 每个候选独立评分，不会遗漏 | 25 次 LLM 调用，CPU 耗时 25s+ |
| 分数可量化比较 | 独立打分无对比参照，精度有限 |
| — | 任意一次超时即触发降级 |

**V2 方案（列表式，当前使用）**

```
候选1..25 → LLM("选出最相关的5个编号") → "3,7,1,12,5"
                                           ↓
                                    按返回编号取 Top-5
```

| 优点 | 缺点 |
|------|------|
| 1 次 LLM 调用，CPU 耗时 ~3s | 候选超过上下文窗口时需截断 |
| LLM 在对比中排序，精度更高 | 返回格式不固定，需正则解析 |
| 单点调用，超时风险低 | 只靠 LLM 主观判断，无可解释分数 |

**决定性考量因素**

选择 V2 方案的核心原因：**单卡 CPU 推理下，25 次调用是不可接受的延迟**。V1 在本地 Ollama 上跑一轮 Reranker 需要 25 秒以上，加上 Embedding 和生成，端到端延迟超过 30 秒，用户体验极差。V2 将 Reranker 延迟压缩到 3 秒以内，总延迟控制在 10 秒左右。

**Bad Case 升级路径**

如果发现列表式 Reranker 频繁出现排序不准（如明显相关的文档被遗漏、无关文档排到前 5），按以下阶梯升级：

| 阶段 | 方案 | 延迟 | 精度 |
|------|------|------|------|
| L1 | **当前**：列表式 LLM 评分 | 3s | 中等 |
| L2 | 加 `num_predict` 限制 + temperature=0，减少随机性 | 3s | 中等+ |
| L3 | 两阶段：先列表式粗筛 25→10，再逐条精排 10→5 | 8s | 高 |
| L4 | 换 Cross-Encoder 模型（BGE-Reranker ONNX）替代 LLM 评分 | 1s | 最高 |
| L5 | Cross-Encoder + LLM 复核：模型打分 Top-8，LLM 最终定 Top-5 | 5s | 最高+ |

> **L4 是终极方案**：Cross-Encoder 专为相关性判断训练，比通用 LLM 更准更快。当前未采用的原因是 CPU 下加载额外 2GB ONNX 模型会与 Qwen 7B 争内存，GPU 机器无此限制。

### 3.3 对话管理

```
首轮: POST /api/kb/:id/chat { question: "年假怎么算？" }
  → 创建 conversation + 2 条 message (user + assistant)

追问: POST /api/kb/:id/chat { question: "那婚假呢？", conversationId: "xxx" }
  → 取最近 6 条历史 → 改写为"婚假怎么算？"
  → 用改写后的问题检索 + 生成
  → conversation.updatedAt 更新
  → conversation.title 自动设为首条问题前 50 字
```

**上下文长度控制**：
- 最近 6 条消息（3 轮对话）作为改写上下文
- LLM 生成时也只传最近 6 条作为对话历史
- 防止长对话撑爆 Qwen2.5-7B 的上下文窗口

## 4. 模块设计

### 4.1 Pipeline 模块

| 文件 | 输入 | 输出 | 关键参数 |
|------|------|------|---------|
| `parser.ts` | 文件路径 | `{ text, fileType }` | 支持 6 种格式（pdf/docx/xlsx/pptx/md/txt） |
| `splitter.ts` | 原始文本 | `TextChunk[]` | chunk_size=500, overlap=50 |
| `embedder.ts` | `string[]` | `number[][]` | 1024 维, 批大小=5, 超时 5min |
| `agent.ts` | 用户问题 | `{ isFunctionCall, result }` | 意图检测 + 时间查询 + 计算器 |
| `generator.ts` | context + question + history | answer string | temperature=0.3, max_tokens=1024 |
| `reranker.ts` | `query + candidates[]` | `ChunkSource[]` | 列表式 LLM 评分, 可选开关 |

### 4.2 Service 模块

| 文件 | 职责 | 依赖 |
|------|------|------|

## 5. 安全模型

### 5.1 认证

```
注册: bcrypt(password, saltRounds=10) → 存 hash，不存明文
登录: bcrypt.compare(password, hash) → JWT sign(access 15min + refresh 7d) → 返回双 token
请求: Authorization: Bearer <access_token> → 中间件 request.jwtVerify() → 解析出 user payload
刷新: 401 → 自动用 refresh_token 换新 access_token（rotation 机制，旧 token 立即删除）
API Key: Authorization: Bearer ak_xxx → authenticateApiKey 中间件 → HMAC-SHA256 哈希比对
```

### 5.2 授权

| 操作 | 权限 |
|------|------|
| 查看知识库 | KB owner 或 member（通过 requireKBAccess 中间件校验 kb_members 表） |
| 编辑知识库 | 仅 owner |
| 删除知识库 | owner 或 admin |
| 上传文档 | KB owner 或 member |
| 删除文档 | KB owner 或 member |
| 访问管理控制台 | 仅 admin |
| 查看对话/导出/反馈 | 仅对话所有者 |

### 5.3 输入校验

| 入口 | 校验方式 |
|------|---------|
| 请求体 | Fastify schema（JSON Schema 定义参数类型和必填项） |
| 环境变量 | Zod parse（启动时自动校验 + 默认值填充） |
| 文件上传 | 扩展名白名单 (.pdf|.docx|.xlsx|.pptx|.md|.txt) + 大小限制 50MB |
| 密码 | 前端校验长度 ≥ 6，后端 bcrypt hash |
| SQL | Drizzle ORM 参数化查询，无 SQL 注入风险 |
| 速率限制 | @fastify/rate-limit 全局 60 req/min |

## 6. 错误处理策略

### 6.1 错误分类

```
AppError (业务错误)
├── NotFoundError (404)    → 资源不存在
├── UnauthorizedError (401) → 未认证
├── ForbiddenError (403)   → 无权限
└── AppError (409)         → 冲突（如重复注册）

FastifyError (框架错误)
└── statusCode → 对应 HTTP 状态码

ZodError (校验错误)
└── 400 → 字段级错误详情

未知错误
└── 500 → 不泄露内部细节
```

### 6.2 响应格式

```
成功:
{ success: true, data: { ... } }

失败:
{ success: false, error: { code: "NOT_FOUND", message: "..." } }

校验失败:
{ success: false, error: { code: "VALIDATION_ERROR", message: "...", details: [...] } }
```

### 6.3 异步处理中的错误

`processDocumentAsync` 的 catch 块：
- 更新 document.status = "error"
- 更新 document.errorMessage = err.message
- 不抛出到上层（fire-and-forget 无接收者）

```
/login              → LoginPage          （无守卫）

/                   → KBListPage         （需登录，否则重定向到 /login）
```

```
  ├─ DocUploader    → @uploaded 事件 → 父组件 refreshKey++ → DocList 重建
  ├─ DocList        → :key="refreshKey" 强制刷新
  ├─ ChatPanel      → 独立状态，接收 :kbId prop
  └─ ConversationList → 独立状态，接收 :kbId prop
```

### 8.3 API 层设计

```typescript
api.ts 封装：
- api.get<T>(url)         → GET 请求
- api.post<T>(url, body)  → POST 请求
- api.patch<T>(url, body) → PATCH 请求
- api.delete(url)         → DELETE 请求（处理 204）

- 从 Pinia store / localStorage 读取 token 注入 Authorization header
- 401 时自动用 refreshToken 静默刷新 accessToken，刷新失败才跳转登录
```

## 9. 部署架构

### 9.1 容器编排

```
docker-compose.yml
├── postgres     (postgres:15-alpine)
│   ├── 暴露 5432
│   ├── 持久化 pgdata volume
│   └── healthcheck: pg_isready
├── redis        (redis:7-alpine)
│   ├── 暴露 6379
│   ├── 持久化 redisdata volume
│   └── healthcheck: redis-cli ping
└── ollama       (ollama/ollama:latest)
    ├── 暴露 11434
    ├── 持久化 ollamadata volume
    └── healthcheck: curl /api/tags
```

**不在 Docker 中的组件**：
- Node.js 后端（`npm run dev` 直接跑在宿主机）
- LanceDB（嵌入 Node.js 进程，数据在 `./data/lancedb/`）
- Vue 前端（Vite dev server 直接跑在宿主机）

**为什么后端不在 Docker 中**：
- 开发阶段热更新更方便
- 调用 Ollama API 走 `localhost`，无容器间网络开销

### 9.2 生产部署建议

```
生产环境:
┌─────────────────────────────────────┐
│  Nginx (HTTPS 终结 + 静态文件)        │
│  /api/* → localhost:3000             │
│  /*     → /var/www/client/dist/      │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│  Node.js (PM2 cluster mode)          │
│  后端服务 × 2 实例                    │
└──────────────────────────────────────┘
               │
    同 docker-compose 的基础设施
    (PostgreSQL + Redis + Ollama)

可选: GPU 云服务器跑 Ollama，CPU 服务器跑其他
```
