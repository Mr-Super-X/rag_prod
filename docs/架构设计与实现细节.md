## 1. 系统概览

### 1.1 产品定位

面向 5-20 人技术团队的内部知识库问答平台。核心价值：上传团队文档 → 自然语言提问 → 基于文档内容精确回答 + 引用来源。

### 1.2 架构层级

```
┌──────────────────────────────────────────────────────────────┐
│                      展示层 (Presentation)                     │
│  Vue 3 SPA  →  登录/注册  →  知识库管理  →  问答交互            │
│  localhost:5173          Vite dev server + 代理到 3000         │
└────────────────────────────┬─────────────────────────────────┘
                             │ HTTP REST (JSON + multipart)
┌────────────────────────────▼─────────────────────────────────┐
│                      编排层 (Orchestration)                    │
│  Fastify Server (port 3000)                                   │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌───────────────┐   │
│  │ Auth 路由 │ │  KB 路由  │ │ Doc 路由  │ │  Chat 路由     │   │
│  │ JWT签发  │ │ CRUD+权限│ │ 上传+异步│ │  问答+对话     │   │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └───────┬───────┘   │
│       └─────────────┴────────────┴───────────────┘           │
│                          │                                    │
│                  Service Layer (无状态，纯逻辑)                 │
│  auth.service / kb.service / doc.service / retrieval.service  │
│  context.service / chat.service                               │
└────────────────────────────┬─────────────────────────────────┘
                             │
        ┌────────────────────┼──────────────────────┐
        ▼                    ▼                      ▼
┌──────────────┐    ┌──────────────┐      ┌──────────────┐
│   持久化层     │    │   AI 推理层   │      │   缓存层      │
│              │    │              │      │              │
│ PostgreSQL   │    │ Ollama       │      │ Redis        │
│ · users      │    │ · /api/chat  │      │ · 会话存储    │
│ · kbs        │    │ · /api/embed │      │ · BM25 倒排   │
│ · documents  │    │              │      │              │

```

### 1.3 关键设计决策

| 决策 | 选择 | 替代方案 | 为什么 |
|------|------|---------|--------|
| 向量库嵌入进程 | LanceDB | Qdrant (Docker) | 国内 Docker 镜像源不稳定，嵌入方案零外部依赖 |
| LLM + Embedding 统一管理 | Ollama | Transformers.js + Ollama | 减少模型服务运行时，降低运维复杂度 |
| 异步文档处理 | Fire-and-forget | 消息队列 | V1 用户少，不需要复杂队列 |
| 前端状态管理 | Pinia | Vuex | Vue 3 官方推荐，更轻量 |
| ORM | Drizzle | Prisma | SQL-like 语法直观，bundle 小，migration 自动生成 |
| JWT 存储位置 | localStorage | Cookie/HttpOnly | V1 简化实现，后续加 refresh token |

## 2. 数据模型

### 2.1 实体关系图

```
users (用户)
  │
  ├── 1:N ── knowledge_bases (知识库)
  │             │
  │             ├── 1:N ── documents (文档)
  │             │             │
  │             │             └── 1:N ── chunks (文本块)
  │             │                         │
  │             │                         └── 1:1 ── LanceDB vectors (向量)
  │             │
  │             └── 1:N ── conversations (对话)
  │                           │
  │                           └── 1:N ── messages (消息)
  │                                       │
  │                                       └── JSONB sources (引用来源)
```

### 2.2 表结构详解

#### users

```
id          UUID    PRIMARY KEY    用户唯一标识
username    VARCHAR(50) UNIQUE    登录用户名
passwordHash VARCHAR(255)         bcrypt 哈希（salt rounds=10）
role        VARCHAR(20)           'admin' | 'user'
createdAt   TIMESTAMPTZ           注册时间
```

**设计说明**：
- 用 UUID 而非自增 ID —— 防止用户枚举，分布式友好
- `role` 存字符串而非外键表 —— 两种角色够用，不引入额外 JOIN

#### knowledge_bases

```
id          UUID    PRIMARY KEY
name        VARCHAR(200)         知识库名称
description TEXT                  可选描述
createdBy   UUID    → users.id   创建者
createdAt   TIMESTAMPTZ
updatedAt   TIMESTAMPTZ
```

**设计说明**：
- `createdBy` 字段实现"我的知识库"隔离 —— 每个用户只看到自己创建的 KB
- V1 不做 KB 级访问控制 —— 任何认证用户可通过 API 访问任何 KB（知道 ID 即可），权限控制在 V2

#### documents

```
id          UUID    PRIMARY KEY
kbId        UUID    → kbs.id (CASCADE)  所属知识库
filename    VARCHAR(500)                原始文件名
fileType    VARCHAR(20)                 pdf|docx|md|txt
fileSize    BIGINT                      字节数
storagePath VARCHAR(500)               磁盘存储路径
status      VARCHAR(20)                 queued|processing|ready|error
chunkCount  INT                        切分后的文本块数
uploadedBy  UUID    → users.id         上传者
errorMessage TEXT                      失败原因
createdAt   TIMESTAMPTZ
updatedAt   TIMESTAMPTZ                最后状态变更时间
```

**设计说明**：
- `status` 状态机：queued → processing → ready | error。前端轮询展示处理进度
- `storagePath` 保存文件物理路径 —— Docker volume 持久化
- `ON DELETE CASCADE` —— 删 KB 自动清理所有文档

#### chunks

```
id              UUID    PRIMARY KEY
docId           UUID    → documents.id (CASCADE)
kbId            UUID    → kbs.id (CASCADE)    冗余字段，避免跨表 JOIN
qdrantPointId   VARCHAR(100)                  对应 LanceDB 向量 ID
content         TEXT                          文本块内容
chunkIndex      INT                           在文档中的序号
metadata        JSONB                         扩展元数据
createdAt       TIMESTAMPTZ
```

**设计说明**：
- `kbId` 故意冗余 —— 检索时按 KB 查 chunks 不需要 JOIN documents，减少一次查询
- `qdrantPointId` 字段名保留（最初选的 Qdrant，后改为 LanceDB）—— 对 PG 无影响，重命名涉及 migration，暂不改

#### conversations

```
id          UUID    PRIMARY KEY
kbId        UUID    → kbs.id (CASCADE)
userId      UUID    → users.id
title       VARCHAR(300)                自动生成（首条消息截取）
createdAt   TIMESTAMPTZ
updatedAt   TIMESTAMPTZ
```

**设计说明**：
- `updatedAt` 用于排序"最近活跃"的对话
- `title` 初始为"新对话"，首轮问答后自动截取用户问题前 50 字

#### messages

```
id              UUID    PRIMARY KEY
conversationId  UUID    → conversations.id (CASCADE)
role            VARCHAR(20)             'user' | 'assistant'
content         TEXT                    消息内容
sources         JSONB                   引用来源数组
createdAt       TIMESTAMPTZ
updatedAt       TIMESTAMPTZ
```

**sources JSONB 结构**：
```json
[
  {
    "chunkId": "doc123_0",
    "content": "引用原文片段",
    "score": 0.032,
    "docFilename": "考勤制度.md"
  }
]
```

### 2.3 LanceDB 向量存储

与 PostgreSQL 的 chunks 表**并行存在**。两者的对应关系：

| PostgreSQL chunks | LanceDB kb_{kbId} |
|-------------------|-------------------|
| `qdrantPointId` | `id` 字段 |
| `content` | `content` 字段 |
| — | `vector` 字段（1024 维浮点数组） |
| `chunkIndex` | `chunkIndex` 字段 |

**不合并的原因**：
- 向量数据不适合存关系数据库（pgvector 千万级以上性能下降）
- LanceDB 专为向量搜索优化（HNSW 索引），比 PG 的 IVFFlat 快一个数量级

## 3. 核心链路详解

### 3.1 文档摄入全链路

```
用户上传文件
  │
  ▼
POST /api/kb/:id/docs (multipart/form-data)
  │
  ├─ Fastify @fastify/multipart 解析
  ├─ 文件扩展名校验 (.pdf|.docx|.md|.txt)
  ├─ 保存到 ./data/uploads/{timestamp}_{filename}
  │
  ▼
processDocument()  ← 立即返回 docId，不阻塞
  │
  ├─ INSERT INTO documents (status="processing")
  │
  └─ processDocumentAsync()  ← 后台执行
       │
       ├─ parseFile()
       │   ├─ .txt/.md → fs.readFile (utf-8)
       │   ├─ .pdf     → pdf-parse (提取文本)
       │   └─ .docx    → mammoth (提取文本)
       │
       ├─ splitText()
       │   ├─ 按段落分割 (\n\s*\n)
       │   ├─ chunk_size=500 字, overlap=50 字
       │   └─ 返回 TextChunk[]
       │
       ├─ embed()
       │   ├─ 每批 5 个 chunk 调用 Ollama /api/embed
       │   ├─ 模型: bge-m3, 输出: 1024 维向量
       │   └─ 超时 5 分钟/批
       │
       ├─ insertVectors()
       │   ├─ LanceDB: 首次创建表, 已存在则追加
       │   ├─ 每条数据: { id, vector, chunkId, docId, kbId, content, chunkIndex, docFilename }
       │   └─ 向量列自动识别为 FixedSizeList(1024)
       │
       ├─ INSERT INTO chunks (逐条)
       │
       ├─ indexBM25()
       │   ├─ @node-rs/jieba 中文分词（失败降级为字符 bigram）
       │   ├─ Redis SADD bm25:{kbId}:{term} {chunkId}
       │   └─ 多词用 Pipeline 批量执行
       │
       └─ UPDATE documents SET status="ready", chunkCount=N
```

**为什么异步处理**：
- CPU 模式下，50KB 文档的 Embedding 需要 1-2 分钟
- 同步处理会让 HTTP 连接挂起直到超时
- 异步方案：响应秒回（仅 INSERT），后台静默处理

### 3.2 检索全链路

```
用户提问: "年假有多少天？"
  │
  ▼
POST /api/kb/:id/chat
  │
  ├─ chat.service.ask()
  │
  ├─ 1. 上下文改写
  │   ├─ 取最近 6 条历史消息
  │   ├─ 调 Ollama /api/chat（temperature=0.1, max_tokens=100）
  │   ├─ 输入: 历史对话 + "那婚假呢？"
  │   ├─ 输出: "公司的婚假政策是什么？"
  │   └─ 超时 2s → 退回原始问题
  │
  ├─ 2. 向量检索 (LanceDB)
  │   ├─ embedSingle(改写后的问题) → 1024 维向量
  │   ├─ LanceDB table.search(vector).limit(25)
  │   └─ 返回 Top-25 + 余弦距离
  │
  ├─ 3. 关键词检索 (BM25)
  │   ├─ @node-rs/jieba 分词 → ["年假", "多少天"]
  │   ├─ 每个词查 Redis: SMEMBERS bm25:{kbId}:{term}
  │   ├─ 统计每个 chunkId 命中多少词
  │   └─ 返回 Top-25 + 命中率分数
  │
  ├─ 4. RRF 融合 (Reciprocal Rank Fusion)
  │   ├─ RRF_score(d) = Σ 1/(k + rank_r(d))
  │   ├─ k = 60（平滑参数）
  │   ├─ 向量排名 + BM25 排名 → 综合分数
  │   └─ 返回 Top-10
  │
  ├─ 5. 兜底判断
  │   └─ sources.length === 0 → "未找到相关内容"
  │
  └─ 6. LLM 生成
      ├─ System Prompt: "基于文档回答，不要编造"
      ├─ User: 检索到的 chunk 内容 + 问题
      ├─ Ollama /api/chat (temperature=0.3, num_predict=1024)
      └─ 返回 { answer, sources }
```

### 3.3 对话管理

```
首轮: POST /api/kb/:id/chat { question: "年假怎么算？" }
  → 创建 conversation + 2 条 message (user + assistant)

追问: POST /api/kb/:id/chat { question: "那婚假呢？", conversationId: "xxx" }
  → 取最近 6 条历史 → 改写为"婚假怎么算？"
  → 用改写后的问题检索 + 生成
  → conversation.updatedAt 更新
  → conversation.title 自动设为首条问题前 50 字
```

**上下文长度控制**：
- 最近 6 条消息（3 轮对话）作为改写上下文
- LLM 生成时也只传最近 6 条作为对话历史
- 防止长对话撑爆 Qwen2.5-7B 的上下文窗口

## 4. 模块设计


## 5. 安全模型

### 5.1 认证

```
注册: bcrypt(password, saltRounds=10) → 存 hash，不存明文
登录: bcrypt.compare(password, hash) → JWT sign → 返回 token
请求: Authorization: Bearer <token> → 中间件 request.jwtVerify() → 解析出 user payload
```

### 5.2 授权

| 操作 | 权限 |
|------|------|
| 查看知识库 | 任何认证用户（知道 KB ID 即可） |
| 编辑知识库 | 仅 owner |
| 删除知识库 | owner 或 admin |
| 上传文档 | 任何认证用户（对任何 KB） |
| 删除文档 | 任何认证用户（对任何文档） |

**V1 已知风险**：
- 任何认证用户可通过猜测 UUID 访问他人的知识库
- JWT 无过期时间，签发后永不过期
- 无 refresh token，无 token 撤销机制
- 无速率限制

### 5.3 输入校验

| 入口 | 校验方式 |
|------|---------|
| 请求体 | Fastify schema（JSON Schema 定义参数类型和必填项） |
| 环境变量 | Zod parse（启动时自动校验 + 默认值填充） |
| 文件上传 | 扩展名白名单 (.pdf|.docx|.md|.txt) + 大小限制 50MB |
| 密码 | 仅前后端各校验长度 ≥ 6（V2 加强度要求） |
| SQL | Drizzle ORM 参数化查询，无 SQL 注入风险 |

## 6. 错误处理策略

### 6.1 错误分类

```
AppError (业务错误)
├── NotFoundError (404)    → 资源不存在
├── UnauthorizedError (401) → 未认证
├── ForbiddenError (403)   → 无权限
└── AppError (409)         → 冲突（如重复注册）

FastifyError (框架错误)
└── statusCode → 对应 HTTP 状态码

ZodError (校验错误)
└── 400 → 字段级错误详情

未知错误
└── 500 → 不泄露内部细节
```

### 6.2 响应格式

```
成功:
{ success: true, data: { ... } }

失败:
{ success: false, error: { code: "NOT_FOUND", message: "..." } }

校验失败:
{ success: false, error: { code: "VALIDATION_ERROR", message: "...", details: [...] } }
```

### 6.3 异步处理中的错误

`processDocumentAsync` 的 catch 块：
- 更新 document.status = "error"
- 更新 document.errorMessage = err.message
- 不抛出到上层（fire-and-forget 无接收者）

## 7. 配置管理

### 7.1 环境变量（Zod 校验）

```typescript
DATABASE_URL    // PostgreSQL 连接串
OLLAMA_URL      // Ollama API 地址，默认 http://localhost:11434
REDIS_URL       // Redis 连接串
JWT_SECRET      // JWT 签名密钥，生产必改
UPLOAD_DIR      // 文件上传目录，默认 ./data/uploads
MAX_FILE_SIZE_MB // 上传限制，默认 50MB
LLM_MODEL       // LLM 模型名，默认 qwen2.5:7b
EMBEDDING_MODEL // Embedding 模型名，默认 bge-m3
EMBEDDING_DIM   // 向量维度，默认 1024
CHUNK_SIZE      // 文本切分大小，默认 500
CHUNK_OVERLAP   // 切分重叠，默认 50
RETRIEVAL_TOP_K // 单路检索返回数，默认 25
RRF_K           // RRF 平滑参数，默认 60
FINAL_TOP_K     // 融合后返回数，默认 10
SIMILARITY_THRESHOLD // 余弦相似度阈值（当前 RRF 模式下未使用）
```

### 7.2 加载顺序

```
1. 读取 .env 文件（手动解析，不依赖 dotenv 包）
2. 以 process.env 现有值为准（不覆盖已设置的环境变量）
3. Zod 校验 + 默认值填充
4. 导出 config 对象（全局单例）
```

### 8.1 路由设计

```
/login              → LoginPage          （无守卫）
/register           → RegisterPage       （无守卫）
/                   → KBListPage         （需登录，否则重定向到 /login）
/kb/:id             → KBDetailPage       （需登录）
```

### 8.2 组件通信

```
KBDetailPage (状态管理)
  ├─ DocUploader    → @uploaded 事件 → 父组件 refreshKey++ → DocList 重建
  ├─ DocList        → :key="refreshKey" 强制刷新
  ├─ ChatPanel      → 独立状态，接收 :kbId prop
  └─ ConversationList → 独立状态，接收 :kbId prop
```

### 8.3 API 层设计

```typescript
api.ts 封装：
- api.get<T>(url)         → GET 请求
- api.post<T>(url, body)  → POST 请求
- api.patch<T>(url, body) → PATCH 请求
- api.delete(url)         → DELETE 请求（处理 204）
- api.upload<T>(url, fd)  → 文件上传（不设 Content-Type）

自动行为：
- 从 localStorage 读取 token 注入 Authorization header
- 非 2xx 响应自动抛出带 message 的 Error
```

```
    ├── 暴露 11434
    ├── 持久化 ollamadata volume
    └── healthcheck: curl /api/tags
```

**不在 Docker 中的组件**：
- Node.js 后端（`npm run dev` 直接跑在宿主机）
- LanceDB（嵌入 Node.js 进程，数据在 `./data/lancedb/`）
- Vue 前端（Vite dev server 直接跑在宿主机）

**为什么后端不在 Docker 中**：
- 开发阶段热更新更方便
- 调用 Ollama API 走 `localhost`，无容器间网络开销

```

               │
```

## 11. V2 扩展指南

### 11.1 替换 LLM 为云端 API

**动机**：CPU 推理慢，接入 DeepSeek/通义千问 API 可实现秒级问答。

**改动范围**：
- `src/pipeline/generator.ts`：新增 `DeepSeekGenerator` 类
- `src/config.ts`：新增 `LLM_PROVIDER` 配置项（ollama | deepseek | qwen）
- 新增 `src/pipeline/llm-factory.ts`：根据配置返回对应生成器

**不改动**：路由、Service 层、前端。

### 11.2 添加 Reranker

**动机**：检索 Top-10 → Reranker 精排 → Top-5，提升准确率。

**改动范围**：
- 新增 `src/pipeline/reranker.ts`：BGE-Reranker-v2-m3 调用封装
- `src/services/retrieval.service.ts`：在 RRF 融合后插入 Reranker 步骤
- `src/config.ts`：新增 `RERANKER_TOP_K` 和 `RERANKER_MODEL`

**注意**：Reranker 模型与 LLM 模型需分时使用 GPU（或 CPU 运行 Reranker）。

### 11.3 多租户数据隔离

**动机**：当前任何用户可访问任何 KB，V2 需要 KB 级权限。

**改动范围**：
- 新增 `kb_members` 表（kbId, userId, role）
- `src/middleware/auth.ts`：新增 `requireKBAccess` 中间件
- 所有 `/api/kb/:id/*` 路由增加 KB 成员校验

### 11.4 消息队列替换

**动机**：文档量大时 fire-and-forget 不可靠。

**改动范围**：
- 引入 BullMQ + Redis（不引入 Kafka 增加运维负担）
- `processDocumentAsync` 改为入队
- 新增 Worker 进程消费队列

### 11.5 前端增强

- 文档处理进度实时推送（SSE / WebSocket）
- 对话内显示思考过程（"正在检索..." → "正在生成..."）
- 打字机效果（逐 token 渲染 Ollama 流式输出）
- 深色模式
- 移动端适配

### 11.6 安全增强

| 项目 | 方案 |
|------|------|
| JWT 过期 | access token 15min + refresh token 7d |
| 密码强度 | 最少 8 位 + 包含字母数字 |
| 速率限制 | @fastify/rate-limit，60 次/分钟 |
| HTTPS | Nginx 反向代理 + Let's Encrypt |
| SQL 注入 | 已通过 Drizzle ORM 规避 |
| XSS | 前端输出内容做 HTML 转义 |
| 文件类型校验 | 检查文件 magic bytes，不仅依赖扩展名 |
